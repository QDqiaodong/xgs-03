# 植物品种对比功能需求说明文档

## 1. 需求目标

在品种列表页提供最多 3 个植物品种的横向对比能力，帮助用户快速比较养护难度和环境要求；同时修复列表页和对比浮层中来自接口的中文乱码问题，确保真实品种数据可直接用于浏览和决策。

## 2. 功能范围

### 2.1 品种对比

- 用户可在品种列表卡片上勾选品种加入对比栏。
- 同时最多选择 3 个品种。
- 选择 2 个及以上时可打开对比浮层。
- 对比浮层按表格横向展示核心养护参数，并对差异项高亮。

### 2.2 对比字段

对比表格展示以下 6 项参数：

| 字段 | 展示名 |
| --- | --- |
| `lightRequirement` | 光照需求 |
| `waterRequirement` | 浇水频率 |
| `temperatureRange` | 适宜温度 |
| `humidity` | 湿度要求 |
| `fertilization` | 施肥建议 |
| `commonDiseases` | 常见病害 |

### 2.3 中文乱码修复

- 修复品种列表页中的品种名、分类、简介、养护参数乱码。
- 修复对比浮层中的品种名、分类和对比参数乱码。
- 保证新初始化环境不会再次导入错码数据。

## 3. 前端实现

### 3.1 解耦结构

前端对比功能拆分为 3 个模块：

- `frontend/src/composables/useCompare.js`
  负责维护对比状态、数量限制、打开关闭浮层。
- `frontend/src/components/CompareBar.vue`
  负责底部对比栏展示与交互发射。
- `frontend/src/components/CompareModal.vue`
  负责对比浮层、横向表格与差异高亮展示。

页面集成位于：

- `frontend/src/views/Categories.vue`

### 3.2 页面交互

- 卡片左上角复选框支持加入/移出对比。
- 已加入对比的卡片带高亮边框和“对比中”标记。
- 底部对比栏展示已选品种，支持单独移除和全部清空。
- 对比浮层中支持继续移除单个品种。
- 差异项通过高亮底色和“差异”标签提示。

## 4. 乱码问题根因

本次乱码并非前端 `axios` 解码问题，也不是 Spring JSON 响应头问题，真实根因有两层：

### 4.1 旧数据已错码入库

`plant_category` 表中的中文字段在历史初始化阶段被以错误字符集写入，导致数据库里已经存在类似 `ç»¿è` 的错码内容。

受影响字段包括：

- `name`
- `category`
- `light_requirement`
- `water_requirement`
- `temperature_range`
- `humidity`
- `fertilization`
- `pruning`
- `common_diseases`
- `description`

同一初始化链路里，`user.nickname` 也存在同类错码。

### 4.2 Redis 缓存保留了旧错码结果

即使数据库数据修正后，`PlantCategoryService` 上的 `plantCategories` 缓存仍会继续返回旧的错码响应，导致接口短时间内看起来“还没修好”。

相关代码：

- `backend/src/main/java/com/plantcare/service/PlantCategoryService.java`

## 5. 修复方案

### 5.1 增加初始化字符集声明

在 `backend/sql/init.sql` 中加入：

```sql
SET NAMES utf8mb4;
```

目的：

- 保证新环境首次执行种子 SQL 时，MySQL 以 `utf8mb4` 正确解析中文文本。

### 5.2 对齐 JDBC 连接字符集

更新以下配置中的数据源连接串，将 `characterEncoding` 统一为 `utf8mb4`，并补充排序规则：

- `backend/src/main/resources/application.yml`
- `docker-compose.yml`

关键参数：

```text
characterEncoding=utf8mb4
connectionCollation=utf8mb4_unicode_ci
```

### 5.3 新增存量数据修复脚本

新增文件：

- `backend/sql/fix_seed_mojibake.sql`

作用：

- 将已错码的 `plant_category` 中文字段批量转回正常 UTF-8 中文。
- 同时修复 `user.nickname`。
- 脚本带错码正则条件，可重复执行，不会对正常中文重复破坏。

### 5.4 清理旧缓存

修复存量数据后，删除 Redis 中的：

```text
plantCategories::all
```

目的：

- 让接口立刻回源数据库，重新生成正常中文缓存。

## 8. 后续维护建议

- 后续若重新初始化数据库，优先使用仓库内最新 `init.sql`。
- 若线上环境已存在历史错码数据，执行 `backend/sql/fix_seed_mojibake.sql` 后，需同步清理 `plantCategories` 缓存。
- 不要再通过前端请求头、`responseEncoding` 或额外 JSON message converter 规避该问题；这类改动不能修复已错码入库的数据。
